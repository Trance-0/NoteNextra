Challenge 1: Reconstruction

  • Minimize reconstruction bandwidth.

Challenge 2: Repair

  • Maintaining data consistency.
  • Failed servers must be repaired:
    • By contacting few other servers (locality, due to geographical constraints).
    • By minimizing bandwidth.

Challenge 3: Storage overhead

  • Minimize space consumption.
  • Minimize redundancy.

Code for storage systems

Naive solution: Replication

Locality is 1 (by copying from another server).

This gives the optimal reconstruction bandwidth.

Use codes to improve storage efficiency

Locality is n−d+1n-d+1n−d+1, high bandwidth.

Parity codes

Let X1,X2,…,Xn∈F2tX_1,X_2,\ldots,X_n\in \mathbb{F}_2^tX1​,X2​,…,Xn​∈F2t​ be the data blocks, take extra server to store the parity.

Reconstruction:

Optimal for reconstruction bandwidth. Only need kkk servers to reconstruct the file.

Overhead:

Only need one additional server

Repair:

Any server failed, reconstruct from the other n−d+1=n−2+1=n−1n-d+1=n-2+1=n-1n−d+1=n−2+1=n−1 servers.

Reed-Solomon codes

Fragment the file X=(X1,…,Xk)X = (X_1, \ldots, X_k)X=(X1​,…,Xk​).

Need 2t≥n2^t\geq n2t≥n servers to store the file.

Reconstruction:

Any kkk servers can reconstruct the file.

Overhead:

Need 2t≥n2^t\geq n2t≥n servers to store the file.

Repair:

Worse, need all servers to reconstruct the file.

Locally Recoverable Codes

Locality: when a node jjj fails,

  • A newcomer node joins the system.
  • The newcomer contacts a “small” number of helper nodes with the message “repairing jjj”.
  • Each of the helper nodes sends something to the newcomer.
  • The newcomer aggregates the responses to find YjY_jYj​.

Notes:

  • No adversarial behavior.
  • No privacy issues.
  • No concern about bandwidth (for now).

Research question:

  • How small can the “small number of nodes” be?
  • How does that affect the rate/minimum distance of the code?
  • How to build codes with this capability?
